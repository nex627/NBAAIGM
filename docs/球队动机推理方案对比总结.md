# 球队动机推理方案对比总结

创建日期：2026-05-24

---

## 背景

本项目的核心目标是让负责交易的AI智能体"更智能"——不仅能生成规则上合法的交易，还能生成**符合球队现实利益和建队逻辑**的交易。

最初的疑问是：是否需要为每支球队创建独立的 Skill？经过多轮讨论，演化出了多个备选方案。本文档总结并对比所有方案，并提供多模型评估框架，供不同模型逐一打分评估。

### 核心约束：NBA是新闻驱动的

NBA的内容本质上是新闻，实时变动，因此方案的**可维护性必须能跟随突发新闻的变动而变动，绝对不能滞后**。具体表现：

1. **球员交易触发级联更新**：一旦球员发生交易，该球员的信息、原球队和新球队的阵容、薪资结构、建队思路都需要同步更新
2. **球队策略可能瞬间翻转**：一笔重磅交易可以让球队从"摆烂"立即转为"争冠"（例如2024年尼克斯得到布里奇斯后策略升级），或反过来
3. **人工跟进是硬性要求**：不存在"写好就不用管"的方案，所有方案都要求人工及时跟进并更新数据，区别仅在于更新的成本和速度
4. **滞后=错误**：如果球队意图数据滞后于现实，AI会基于过时信息生成荒谬交易，比没有意图数据更危险

## 方案0：30个球队Skill

### 核心思路
为NBA 30支球队每队创建一个独立的Skill模块，每个Skill包含该球队的专属知识、建队逻辑、总经理风格等。

### 具体实现
- 每个Skill是一个独立的子代理或工具函数
- 每个Skill有自己的系统提示词和知识库
- 交易评估时调用对应球队的Skill

---

## 方案1：球队状态档案 + 统一动机推理层（JSON）

### 核心思路
不按球队拆Skill，而是建立结构化的球队知识库，让智能体在处理交易前主动查询相关球队的档案。

### 具体实现
```json
{
  "lac": {
    "stage": "战略性退却",
    "stage_why": "2027年薪资大清洗后重获主动权，无自有首轮不能摆烂",
    "hard_rules": ["不能送出首轮签", "不能换回30岁以上即战力"],
    "prefer_return": ["年轻球星", "优质选秀权"],
    "gm_style": "弗兰克保守型操作，但鲍尔默激进且财力无限",
    "horizon": "2027-28赛季"
  }
}
```

- **第一步**：每队定义结构化档案（JSON格式，6大字段）
- **第二步**：创建通用查询工具 `get_team_profile(team_name)`
- **第三步**：系统提示词注入动机推理指令

## 方案2：6原型 + 30标签（MD）

### 核心思路
30支球队的建队逻辑本质上只有6种原型，每队用一行标签指向对应原型。

### 具体实现
```markdown
## 推理原型

### 争冠冲刺型
窗口开着，愿意付出未来换即战力。可以送出首轮末，但首轮中和前段要慎重。
偏好：明确补强短板的即战力，到期合同可接受。

### 战略性退却型
不摆烂但变现老将，为下一个周期蓄力。可以交易核心老球星，目标换回年轻资产。
偏好：年轻球星、优质选秀权、有性价比的拼图。
铁律：不送出首轮签，不换回30岁以上即战力。

（...其余4种原型...）

### LAC · 快船
原型：退却型 | 2027年薪资大清洗后重获主动权，无自有首轮不能摆烂 | 弗兰克保守/鲍尔默激进 | 时间线：2027-28

### OKC · 雷霆
原型：争冠型 | SGA+杰威+切特核心成型，选秀权储备历史级 | 普雷斯蒂极度精算 | 时间线：当下-2028
```

- **6段原型逻辑**：覆盖所有球队的建队模式
- **30行标签**：每队一行，指向原型+补充关键点

## 方案5：30句话 + 硬规则 + 推理方法论（MD数据 + 系统指令）

### 核心思路

方案0-4全都在解决同一个问题：**给LLM什么信息**。但没人问：**LLM怎么用这些信息？**

方案4给了快船"不摆烂但变现老将 ❌不送首轮 ❌不换回30+即战力"。LLM看到了。但它真的会用吗？在不同运行中，LLM可能：
- 扫一眼就过去了（没触发注意力）
- 在10条推理之后忘记了（上下文窗口竞争）
- 被"合理性推理"说服自己跳过约束（"虽然写了❌不送首轮，但这笔交易回报太好了"）

方案5补上缺失的一环：**在系统提示词中注入一套推理协议，明确告诉LLM如何利用意图数据。** 数据文件完全复刻方案4，只增加一个~15行的静态系统指令块。

### 具体实现

**数据文件（与方案4完全一致，50行MD）**：

```markdown
# 球队当前意图与硬规则（极高优先级）

- 亚特兰大老鹰：囤积年轻天赋，可交易 veteran 换选秀权
- 波士顿凯尔特人：争冠窗口全开，愿意付出未来换即战力
- 布鲁克林篮网：重建中，收集选秀权和年轻球员 ❌不送首轮
- 洛杉矶快船：不摆烂但变现老将，为2027年蓄力 ❌不送首轮 ❌不换回30+即战力
...（共30行）
```

**系统提示词注入（新增~15行静态指令，方案4没有的部分）**：

```markdown
## 球队意图推理协议（在处理任何交易时强制遵循）

对于接收到的每笔交易提案，你必须按以下步骤逐一执行：

### 步骤1：识别双方
确定这笔交易涉及的**两支球队**（A队和B队）。

### 步骤2：检查硬规则（阻塞性）
从球队意图列表中，定位A队和B队的❌项。
逐条检查交易是否违反任一❌项：
- 若违反 → **立即拒绝交易**，在解释中引用具体被违反的❌项
- 若未违反 → 继续步骤3

### 步骤3：意图对齐检查
对照A队和B队的一句话意图，评估：
- 这笔交易是否符合各自球队的战略方向？
- 若两队意图存在天然矛盾（如一队争冠一队重建），这笔交易是否合理地调和了这一矛盾？
- 如果不合理 → **建议修改或拒绝**

### 步骤4：综合判断
结合薪资匹配、球员价值等硬数据，给出最终建议。
```

### 设计要点

- **推理协议是静态的**：一旦写好几乎不需要更新，不与NBA新闻联动
- **数据文件完全复刻方案4**：数据维护量、实时可维护性完全相同
- **推理协议强化❌规则的约束力**：从"LLM可能会看到"升级为"LLM被要求逐条验证"
- **提升行为一致性**：多次运行同一交易，有推理协议比没有协议的输出更稳定

### 与前5个方案的本质区别

| 维度 | 方案0-4 | 方案5 |
|------|---------|-------|
| 解决的问题 | 给LLM**什么**信息 | 给LLM**什么**信息 + 告诉LLM**如何**使用 |
| 改进方向 | 数据格式、数据来源、数据结构 | 推理流程、LLM行为约束 |
| 对LLM的假设 | 给数据后LLM自己知道怎么用 | LLM需要明确的推理步骤才能用好数据 |
| 新增维护项 | 数据文件 | 无（推理协议是静态的） |

## 方案6：球队档案报告（MD）—— 金融分析师模式

### 核心思路

方案3-5的"一句话"模式在评估中得分最高，但存在根本性缺陷：**NBA交易评估是信息完整性问题，不是信息压缩问题。**

一句话"快船不摆烂但变现老将"丢失了：
- 小卡的膝盖状况和合同细节
- 鲍尔默的财务底线和投资意愿
- 弗兰克（GM）的保守交易风格
- 快船与哪些球队有交易历史、与哪些球队从不交易
- 球队距离第二土豪线的具体距离
- 2027年薪资空间释放的具体时间线

这些信息在评估一笔交易是否可行时都是决定性的。缺少任何一个，LLM都可能生成"大差不差但差了十万八千里"的交易建议。

方案6采用**金融分析师报告模式**：每队一份结构化多维度MD档案，像金融分析师评估一家公司一样评估一支球队。

### 最终设计：9章节 + 分层时效标注 + 全量注入

**每队一份MD档案，包含9个固定章节：**

```markdown
# [缩写] [球队全名]

> 最后更新：YYYY-MM-DD

## 1. 建队阶段与战略方向
> ⏱ 更新频率：低 | 上次验证：YYYY-MM-DD

[争冠/重建/过渡/摆烂，时间线，核心策略]

## 2. 核心球员状态
> ⏱ 更新频率：高 | 上次验证：YYYY-MM-DD

[每个核心球员拆为3个子字段：]
- **球员名**：年龄，合同状况
  📋 当前状态：[健康/受伤/复出中/赛季报销等，附具体日期]
  ⚠️ 长期风险：[慢性伤病、衰退趋势等较稳定的风险判断]
  💰 交易价值：[高/中/低 + 原因]

## 3. 总经理决策风格
> ⏱ 更新频率：极低 | 上次验证：YYYY-MM-DD

[保守/激进、交易习惯、历史案例、关键偏好]

## 4. 老板意愿与投资哲学
> ⏱ 更新频率：低 | 上次验证：YYYY-MM-DD

[是否越级GM、花钱意愿、对输球的容忍度、长期承诺]

## 5. 教练体系与战术偏好
> ⏱ 更新频率：极低 | 上次验证：YYYY-MM-DD

[偏好什么类型球员、战术风格、对阵容构建的影响]

## 6. 薪资状况与约束
> ⏱ 更新频率：中 | 上次验证：YYYY-MM-DD

[具体数字、土豪线距离、操作空间、关键时间节点]

## 7. 选秀权状况
> ⏱ 更新频率：中 | 上次验证：YYYY-MM-DD

[持有哪些、欠哪些、保护条款、交易价值]

## 8. 交易偏好
```

### 时效性分层标注

9个章节的信息腐烂速度差异极大，需要分层标注：

| 章节 | 腐烂速度 | 典型更新频率 | 说明 |
|------|---------|------------|------|
| 1. 建队阶段 | 慢 | 1-2年 | 战略方向很少突变 |
| 2. 核心球员状态 | **极快** | 每周/每月 | 伤病、状态变化最快 |
| 3. GM决策风格 | 极慢 | 仅GM更换时 | 人的性格几乎不变 |
| 4. 老板意愿 | 慢 | 1-2年 | 投资哲学稳定 |
| 5. 教练体系 | 极慢 | 仅教练更换时 | 战术体系稳定 |
| 6. 薪资状况 | 中 | 交易截止日+自由市场 | 每年2-3次大变动 |
| 7. 选秀权 | 中 | 交易截止日+选秀夜 | 每年2-3次大变动 |
| 8. 交易偏好 | 中 | 随战绩变化 | 赛季中可能调整 |
| 9. 交易关系网 | 慢 | 1-2年 | 关系网变化缓慢 |

**第2章（核心球员状态）特殊处理**——拆为3个子字段：
- 📋 **当前状态**：最易腐烂，需频繁更新（伤病、复出、状态起伏）
- ⚠️ **长期风险**：较稳定（慢性伤病、年龄衰退趋势）
- 💰 **交易价值**：中速腐烂（随市场行情和球员表现变化）

每章标注 `⏱ 更新频率` 和 `上次验证` 日期，LLM可根据验证日期判断信息可信度。

### 注入策略：全量注入

**30支球队的档案全部注入prompt，不做选择性截取。**

理由：
1. 多队交易可能涉及3-4队甚至更多，无法预判需要哪些球队
2. 即使不直接参与的球队，LLM也需要知道"交易后对其他球队的影响"
3. 准确性优先，token消耗不是问题
4. 代码更简单，不需要选择性注入逻辑

### 信息重叠处理

档案和代码已有数据（薪资、球员、PER等）会有重叠，但**不删减**。档案补充的是代码没有的定性信息：
- 代码说"伦纳德薪资52M" → 档案补充"膝盖慢性问题，出勤率极低，交易价值被伤病大幅压价"
- 代码说"快船超税" → 档案补充"距离第二土豪线370万，不能聚合薪资"
- 代码说"首轮签0个" → 档案补充"2025和2027首轮在雷霆手中，摆烂无意义"

冗余不是问题，信息缺失才是问题。

### 为什么选择MD而非JSON

方案1用JSON，方案6用MD。原因：
1. **LLM理解效率**：自然语言MD比JSON高15-25%
2. **手动编辑友好**：MD无格式陷阱，JSON一个逗号错误就全局瘫痪
3. **表达自由度**：伤病隐患、GM风格等主观信息用自然语言比用JSON字段更准确
4. **章节结构已经足够**：固定9个章节提供了结构化思维框架，不需要JSON的强制格式

## 多模型评估框架

### 评估目标

通过多个不同模型对6种方案进行独立评估，确保最终选择的方案：
1. **有效**：方案落地后，LLM能真正利用这些信息生成更合理的交易建议
2. **准确**：符合NBA现实交易逻辑，能真实提升AI交易建议的质量
3. **实时可维护**：能跟随突发新闻快速更新，滞后成本最低

### 评估维度（每个方案统一使用这3个维度）

| 维度 | 定义 | 权重 |
|------|------|:---:|
| **有效性** | 方案落地后，LLM能否真正利用这些信息生成更合理的交易建议。包括：LLM对注入信息的理解程度、信息对推理的实际影响力、方案对交易质量的实际提升幅度 | 25% |
| **准确性** | 方案能否准确反映NBA球队的真实建队逻辑和交易动机。**包括：伤病隐患、GM决策偏好、薪资约束细节、交易关系网、核心球员状态等所有影响交易决策的维度** | 50% |
| **实时可维护性** | 当突发新闻发生时（球员交易、球队策略翻转、管理层变动等），方案能多快被更新到反映最新现实。包括：单次更新的改动量、更新出错的风险、级联更新的复杂度、人工跟进的成本 | 25% |

### 评分标准（每个维度10分制）

| 分数 | 说明 |
|:---:|------|
| 9-10 | 非常优秀，几乎没有缺陷，完全符合要求 |
| 7-8 | 良好，有少量小问题，但整体质量很高 |
| 5-6 | 中等，有明显缺陷，但核心思路可行 |
| 3-4 | 较差，有较多问题，需要大量修改 |
| 1-2 | 很差，存在根本性缺陷，不建议采用 |

### 实时可维护性专项评分指引

评估"实时可维护性"时，必须用以下突发场景来检验每个方案：

| 突发场景 | 检验内容 |
|----------|---------|
| **场景A：重磅球员交易** | 球星X从A队交易到B队，A队从争冠变重建，B队从重建变争冠。方案需要更新哪些内容？改动几处？ |
| **场景B：总经理更换** | A队更换总经理，交易风格从保守变激进。方案需要更新哪些内容？改动几处？ |
| **场景C：赛季中伤病潮** | A队核心球员赛季报销，策略从争冠转为保守。方案需要更新哪些内容？改动几处？ |
| **场景D：休赛期多笔交易** | 一周内发生5笔涉及10支球队的交易。方案需要更新哪些内容？总改动量多大？ |

对每个场景，评估者需要回答：
- 需要修改几个文件/几处内容？
- 修改是否会引发级联影响（改一处需要连带改其他地方）？
- 修改出错的风险有多大？
- 从看到新闻到完成更新，预计需要多长时间？

### 评估方法

1. 每个模型独立对6个方案进行评估
2. 对每个方案，分别从**有效性**、**准确性**、**实时可维护性**三个维度打分
3. 每个维度打分后，必须写出**3条具体的优点**和**3条具体的缺点**
4. 评估"实时可维护性"时，必须用上述4个突发场景逐一检验
5. 计算加权总分：总分 = 有效性×0.25 + 准确性×0.50 + 实时可维护性×0.25
6. 在评估结果中标注**模型名称**和**评估日期**

**评估模型**: GLM-5.1
**评估日期**: 2026-05-24

### 优点
1. 信息密度最高——每队独立Skill可以包含该队所有维度的深度知识（历史交易模式、GM性格、老板偏好、更衣室文化等），其他方案都无法做到这个粒度

## 关键洞见

### 1. 评估框架的根本性错误：信息完整性 ≠ 信息压缩

**这是整个评估过程中最大的盲点。** 三个LLM模型一致给方案3/4/5打了最高分，但这恰恰暴露了LLM的系统性偏见——**LLM天然认为"简洁=高效"**，因为训练目标之一就是信息压缩。

但NBA交易评估不是信息压缩问题，是信息完整性问题。就像金融分析师不会用一句话来评估一家公司，NBA交易也不应该用一句话来评估一支球队。

一句话"快船不摆烂但变现老将"丢失了：
- 小卡的膝盖状况和合同到期细节
- 鲍尔默的财务底线和投资意愿
- 弗兰克（GM）的保守交易风格
- 快船与哪些球队有交易历史
- 球队距离第二土豪线的具体距离
- 2027年薪资空间释放的具体时间线

这些信息在评估一笔交易是否可行时都是决定性的。缺少任何一个，LLM都可能生成"大差不差但差了十万八千里"的交易建议。

### 2. 准确性是生死线，维护成本是可以接受的代价

之前的评估框架把"实时可维护性"和"准确性"放在同等重要的位置（各35%），这个隐含假设是"维护成本和准确性同等重要"。但实际需求恰恰相反——**准确性是生死线，维护成本是可接受的代价**。

宁可每次新闻发生后花1小时更新档案，也不要用一句话给一个"大差不差"的结果。因为"大差不差"在交易评估中意味着"差了十万八千里"。

### 3. 方案6（档案报告）是唯一符合"金融分析师标准"的方案

如果用金融分析师的标准来衡量：
- 一句话报告（方案3/4/5）→ 准确性不可能超过3分
- 6原型分类（方案2）→ 准确性不可能超过4分
- 结构化档案报告（方案6）→ 准确性可以达到9分

方案6的6个固定章节（建队方向、核心球员状态、GM风格、薪资约束、交易偏好、交易关系网）覆盖了所有影响交易决策的关键维度。这是唯一一个能让LLM像金融分析师一样全面评估球队的方案。

### 4. 方案5的高分是"推理效率"的高分，不是"准确性"的高分

方案5的推理协议确实解决了"LLM如何使用信息"的问题，这是维度层面的突破。但方案5的数据基础仍然是"一句话+❌标记"，信息量没有增加。推理协议让LLM更高效地使用了有限的信息，但无法弥补信息本身的不足。

**推理效率 × 信息完整性 = 最终准确性**

方案5的推理效率是9分，但信息完整性只有3-4分（一句话），最终准确性 = 9 × 0.4 = 3.6分。
方案6的推理效率是7分（无推理协议），但信息完整性是9分，最终准确性 = 7 × 0.9 = 6.3分。

### 5. 最优方案应该是方案6 + 方案5的推理协议

方案6解决了"信息完整性"问题，方案5解决了"推理效率"问题。两者结合才是最优解：

```
方案6（档案报告数据）+ 方案5（推理协议）= 方案7（待定义）
```

每队一份多维度MD档案 + 系统提示词中的结构化推理协议。信息最完整 + 推理最系统。

### 6. LLM缺的不是数据，是态度
你现有的代码已经提供了足够的硬数据（薪资、球员、合同、选秀权）。LLM缺的是**对这些数据的立场**——
- 快船看到没首轮，LLM知道事实，但不知道快船"认命"还是"想赎回"
- 快船看到伦纳德35岁，LLM知道事实，但不知道快船"想卖"还是"想留"

所有方案本质上都是在补充这个"态度"。但方案3/4/5的"态度"太粗略了，方案6的"态度"才够具体。

### 7. 当前架构约束：没有Function Calling
方案1中提到的"调用 `get_team_profile` 工具"在当前架构下无法实现。当前项目是纯prompt驱动，LLM无法"调用工具"。所有档案数据必须**预构建进prompt**，不能让LLM查询。

方案6的token占用（~6000-9000 tokens）是一个现实约束，但可以通过**选择性注入**解决：只注入交易涉及的2支球队的档案，而不是全部30支。

### 8. NBA是新闻驱动的，滞后=错误
NBA的内容本质上是新闻，实时变动。一笔重磅交易可以让球队策略瞬间翻转。因此：
- **不存在"写好就不用管"的方案**——所有方案都要求人工及时跟进并更新数据
- **滞后比没有更危险**——基于过时意图数据生成的交易建议，比完全没有意图数据更荒谬
- **但维护成本不应主导方案选择**——宁可多花时间维护，也要保证准确性

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|:---:|---------|
| 档案信息不准确 | 高 | 先写热门球队，冷门球队后续校准；每个章节标注信息来源 |
| 档案过时 | 高 | 加 `last_updated` 标记，超过2周标注警告；突发新闻后强制更新 |
| LLM忽略档案中的❌规则 | 中 | 放在prompt最前面，系统提示词强调优先级；必要时加推理协议 |
| Token占用过大 | 中 | 选择性注入：只注入交易涉及的2支球队档案，不注入全部30支 |
| 初始创建成本高 | 中 | 2-3天完成初版；可先做10支热门球队，后续补充 |
| 模型评估存在"信息压缩偏见" | 高 | 已识别，权重已调整；准确性回归现实后方案6排名第一 |
| 突发交易后更新滞后 | 高 | 建立更新SOP：看到新闻→1小时内更新相关球队档案→验证 |
